# StitchMeIn — "Stitch Yourself In" Prototype

A demo prototype built for a TikTok design challenge. The central feature is **Stitch Yourself In** — an AI-powered creation tool that lets users insert themselves into any video they love. Users pick a moment from a video, take a selfie, describe an action, and the app composites them into the scene and animates it into a seamless video clip that posts back to their feed.

## Key Features

**Stitch Yourself In Flow**

- One-tap entry from any paused feed video
- **Clip Selector** — draggable filmstrip with bracket window; initialises at the exact frame where the user paused
- **Character Capture** — in-browser selfie via `getUserMedia`, with re-take support
- **AI Prompt Sheet** — scene-aware idea chips generated by Gemini from the selected frame, plus free-text input

**Generation Pipeline**

- Stage 1: Gemini composites the user into the scene as a static image — shown immediately as a preview while video renders
- Stage 2: Video engine (Veo or Kling) animates the composite into a smooth 5–8 second clip
- Animated loading screen with per-stage progress and orbiting icons

**Preview & Post**

- Crossfade from original clip into the AI-generated video
- Background music carries continuously across the transition
- Post screen with caption editing, hashtag chips, and mock share options

**FYP Feed (supporting context)**

- Vertical snap-scroll feed with swipe gesture (mouse drag / touch)
- Autoplay / pause with tap-to-toggle
- Full-bleed 9:16 video with sidebar actions (like, comment, save, share)
- Newly stitched posts land at the top of the feed; playback loops source clip → AI segment

## Technical Challenges & Solutions

### Smooth video stitching

The core challenge is making the transition from the original clip into the AI-generated segment feel continuous rather than jarring.

**Approach:** instead of generating a video directly from a text prompt, the pipeline uses a two-keyframe interpolation strategy:

1. The **last frame of the user's selected clip** is extracted and sent to Gemini as the scene anchor
2. Gemini composites the user into that exact frame, producing an **end-keyframe image**
3. The video engine (Veo / Kling) receives both the original last frame as the _start_ and the composited image as the _end_, with instructions to animate the transition between them

This means the generated clip begins at the precise visual state where the source video ended, making the stitch look intentional rather than pasted.

### Intermediate image as progress signal

AI video generation takes 1–3 minutes. To prevent a blank loading screen and set user expectations, the Gemini composite image is surfaced immediately (Stage 1 complete, ~10–20s) while the video engine continues working in the background. Users see _their result_ before the animated version is ready, which reduces perceived wait time and communicates that generation is on track.

### Prompting video models for a clean cut

Current video models have a strong default tendency to **crossfade** between the start and end frames rather than producing a true continuous motion. Left unconstrained, the output looks like a dissolve transition rather than a person appearing in a scene.

The prompts sent to both Veo and Kling explicitly counter this by specifying single-shot camera behaviour:

- **Veo:** `"One-shot handheld camera movement from start frame to target frame. Match the lighting and maintain photorealistic detail throughout."`
- **Kling:** `"Cinematic one-shot. The video must begin exactly at the start frame and resolve perfectly into the target frame. No identity drift. Continuous handheld dolly-in motion."`

The "one-shot" framing discourages the model from treating the transition as a cut or dissolve.

### Face consistency across generations

Gemini's image model tends to drift from the selfie's likeness when compositing against a complex background. To anchor face identity, a dedicated reference photo (`user.jpg`) is pre-loaded at server start and sent to every generation request as a high-priority identity reference, separate from the live selfie. The selfie is then used only for pose, expression, and lighting cues. This two-image approach significantly reduces identity drift at no extra latency cost.

### API key security

All three API keys (Gemini, Veo, Kling) are held server-side only. The frontend never sees them — it calls local Express endpoints which proxy to the upstream APIs. For Veo specifically, GCS-signed video URIs also require the key at download time, so the `/api/video-proxy` endpoint streams the authenticated video bytes directly to the browser.

## Cost & Latency Considerations

**Dev mock mode** (`USE_DEV_MOCK = true` in `server.js`)
Skips all real AI calls and returns local placeholder assets. Stage timings are individually configurable (`MOCK_STAGE_MS = [3000, 5000, 8000]` ms per stage) to simulate realistic generation pacing during demos without incurring API costs.

**Dual video engine**
`VIDEO_ENGINE` switches between `"VEO"` and `"KLING"` without any frontend changes. Veo generates 8-second 9:16 clips (billed per second of output, ~2–3 min). Kling generates 5-second clips (billed per task, ~1–2 min). Kling is faster and cheaper for demos; Veo produces higher-quality motion.

**Request body limit**
Express is configured for 50 MB JSON bodies to accommodate multiple base64-encoded video frames and images in a single request without chunking.

## Areas to Improve

- **Clip selection UX** — the current 8-frame bracket window always selects a fixed duration. A production version would let users drag the bracket edges to choose any clip length, with the selected duration shown in real time.
- **Video continuity at the stitch point** — the crossfade from source clip to AI video is an opacity transition. A tighter approach would blend the last source frame with the first AI frame at the pixel level (e.g. optical flow morphing) to make the cut invisible.
- **Face consistency** — identity drift is reduced by the reference image but not eliminated. Fine-tuning or a dedicated face-swap post-processing step would make likeness more reliable across varied scene lighting.
- **Generation time feedback** — the loading screen shows estimated minutes but no live countdown or progress percentage. Surfacing real-time polling status (e.g. Veo operation progress) would reduce uncertainty during the wait.
- **Video engine fallback** — if the primary video engine (Veo/Kling) fails, the flow currently surfaces an error. A graceful fallback — such as animating the composite image with a Ken Burns effect — would keep the experience intact even on API failure.
- **Post-generation editing** — users cannot adjust the result (re-crop, re-prompt, trim) before posting. Adding lightweight editing steps between the preview and post screens would reduce regret after publishing.
- **Persistent feed** — generated posts live only in React state and disappear on refresh. Persisting them to a backend would let users revisit their creations and share real URLs.

## Setup & Installation

### Prerequisites

- Node.js 18+
- A `.env` file in the project root (see below)

### 1. Install dependencies

```bash
npm install
```

### 2. Configure environment variables

Create a `.env` file in the project root:

```env
# Required for Gemini image generation and Veo video generation
GEMINI_API_KEY=your_gemini_api_key

# Required only if VIDEO_ENGINE = "KLING"
KLING_ACCESS_KEY=your_kling_access_key
KLING_SECRET_KEY=your_kling_secret_key
```

### 3. Add placeholder assets (dev mode)

```
public/
  placeholders/
    mock_remix.jpg    ← any JPEG (returned as the "generated" image)
    mock_video.mp4    ← any 9:16 MP4 (returned as the "generated" video)
  images/
    user.jpg          ← your face reference photo (improves likeness in live mode)
```

### 4. Add feed videos

```
public/videos/video1.mp4
public/videos/video2.mp4
public/videos/video3.mp4
public/videos/video4.mp4
```

### 5. Run the app

```bash
# Start both Vite (port 5173) and Express (port 3001) together
npm run dev:all

# Or start them separately
npm run dev      # Vite frontend
npm run server   # Express backend
```

Open [http://localhost:5173](http://localhost:5173) in a browser. For the best experience, open DevTools and set the viewport to **390 × 844 px** (iPhone 14).

### 6. Toggle dev / live mode

In `server.js`:

```js
const USE_DEV_MOCK = true; // false = real AI, true = placeholder assets
const VIDEO_ENGINE = "VEO"; // "VEO" | "KLING"
```
